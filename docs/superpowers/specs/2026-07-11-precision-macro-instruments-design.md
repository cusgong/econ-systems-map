# 정밀 거시경제 계기 3D 시스템 설계

> 작성일: 2026-07-11  
> 상태: 서면 검토 요청  
> 제품명: Econ Systems Map  
> 디자인 방향: **A2, Precision Macro Instruments**  
> 제작 방식: **Blender 개별 조형 + 단일 glTF 바이너리(GLB) 라이브러리 + Three.js 상태 제어**

## 1. 결정 요약

이 개편은 30개 변수를 같은 구체의 색상 변형으로 보여 주는 방식을 끝내고, 각 변수를 서로 다른 전문적인 3D 계기 오브제로 바꾼다.

- **형태**는 변수의 정체성을 말한다. 30개 모두 다른 실루엣을 갖는다.
- **범주색**은 역할을 묶는다. 전체 도색 대신 인레이, 눈금, 렌즈, 내부광 등 표면의 10~20%에만 쓴다.
- **크기**는 강한 직·간접 인과관계를 함께 반영한 허브성을 말한다.
- **위치**는 현재 공간 문법을 그대로 보존한다. 높이는 층위, 반경은 국내·글로벌 거리, 각도는 경제 부문이다.
- **움직임**은 상태 변화를 말한다. 기본 상태는 정지하고, 선택·전파 도착·시뮬레이션 반응 때만 짧게 작동한다.
- **텍스트**는 선명한 2D 라벨로 유지한다. 모델 안에 작은 글자나 숫자를 굽지 않는다.

구현은 먼저 대표 변수 6개를 실제 앱 안에서 검증한 뒤, 같은 조형 문법으로 나머지 24개를 확장한다. Blender 설치는 이 서면 스펙 승인 직후 진행한다.

## 2. 문제 정의와 근거

현재 노드는 모두 하나의 `SphereGeometry`를 공유하며, `MeshBasicMaterial`, 와이어프레임, 가산식 광륜으로 차이를 만든다. 이 재질은 장면 조명에 반응하지 않기 때문에 면의 곡률, 모서리, 표면 깊이가 빛으로 드러나지 않는다. 따라서 프리미티브를 더 많이 조합하는 것만으로는 전문적인 3D 품질에 도달하기 어렵다.

현재 크기는 직접 연결 수만 사용한다.

```js
size = 1.35 + Math.min(1.15, degree * 0.085)
```

이 값은 사용자가 선택한 의미인 **인과 연결 허브성**과 다르다. 간접 전파력, 연결 강도, 근거 확실성을 포함하는 별도 점수가 필요하다.

## 3. 목표와 비목표

### 목표

- 30개 변수를 흑백 실루엣만으로도 서로 구별할 수 있게 한다.
- 모든 오브제가 하나의 고급 정밀기기 제품군처럼 보이게 한다.
- 형태, 색, 크기, 위치, 움직임이 서로 다른 정보를 담당하게 한다.
- 기존 탐색, 시뮬레이터, 사례, 루프, 레버 드래그 기능을 보존한다.
- 정적 배포, 오프라인 로딩, 로컬 Three.js r170 구조를 유지한다.
- 중급 모바일에서도 상호작용 가능한 프레임 속도를 확보한다.

### 비목표

- 변수마다 사실적인 사물 미니어처를 만드는 작업은 아니다.
- 집, 물방울, 동전, 사람 모형 같은 클립아트식 직역을 목표로 하지 않는다.
- 장면 전체를 네온 홀로그램이나 게임 인터페이스로 바꾸지 않는다.
- 카테고리마다 같은 모델을 재사용하지 않는다.
- 모델 안의 장식으로 새로운 경제 지표를 추가하지 않는다.
- 실시간 광선 추적, 무거운 투명 전송재, 상시 블룸을 도입하지 않는다.

## 5. 시각 언어

### 5.1 조형 문법

전체 오브제는 다음 공통 규칙으로 한 제품군을 이룬다.

- 주 실루엣은 원통, 프레임, 짐벌, 베인, 매니폴드, 격자처럼 정밀기기에서 가져온다.
- 모든 노출 모서리에 일관된 작은 베벨을 둔다. 베벨 폭은 정규화된 모델 지름의 약 1.5~3%다.
- 큰 덩어리 1개, 보조 구조 1~2개, 범주색 악센트 1개로 읽히게 한다.
- 장식용 나사, 미세 홈, 작은 문자처럼 노드 표시 크기에서 사라지는 디테일은 만들지 않는다.
- 앞·옆·위 3방향에서 실루엣이 무너지지 않게 한다.
- 좌우 대칭을 기본으로 하되, 변수 의미가 흐름·격차·긴장을 포함할 때만 통제된 비대칭을 쓴다.

### 5.2 재질 체계

물리 기반 렌더링(PBR)은 `MeshStandardMaterial`을 기본으로 한다. `MeshPhysicalMaterial`은 대표 모델에서 렌즈가 반드시 필요한 경우에만 제한적으로 쓴다.

| 재질 | 역할 | 목표 인상 |
|---|---|---|
| Dark Titanium | 주 몸체 | 어두운 티타늄, 넓고 부드러운 반사 |
| Satin Alloy | 프레임·축 | 차가운 새틴 금속, 베벨 강조 |
| Technical Ceramic | 절연부·기준판 | 낮은 금속성, 단정한 회백색 |
| Smoked Lens | 제한적 렌즈 | 내부를 완전히 드러내지 않는 스모크 표면 |
| Category Inlay | 역할 악센트 | 현재 9개 범주색, 표면의 10~20% |

범주색은 현재 `data/nodes.js`의 색을 정본으로 사용한다. 카테고리 인레이는 선택·시나리오 상태에서도 본래 색을 잃지 않는다.

|---|---|
| 정책 | `#35d0b0` |
| 통화·금융 | `#4fd8ff` |
| 자산시장 | `#9d8cff` |
| 시장 심리 | `#ff9ecb` |
| 실물경제 | `#6fe38a` |
| 물가·임금 | `#ff7666` |
| 원자재 | `#ffc857` |
| 외생 변수 | `#c678ff` |
| 대외·환율 | `#5f8dff` |

### 5.3 조명과 톤

- 색 공간은 `SRGBColorSpace`를 사용한다.
- 톤 매핑은 영화형 명암 압축(ACES Filmic)을 사용한다.
- `RoomEnvironment + PMREM`으로 공통 환경 반사를 만든다.
- 넓은 중성 키라이트와 약한 차가운 림라이트를 둔다.
- 모델이 떠 있는 지도이므로 바닥에 떨어지는 실시간 그림자는 기본적으로 끈다.
- 기본 발광은 거의 0으로 두고, 선택·전파 상태의 악센트에만 짧고 약하게 쓴다.
- 후처리 블룸은 1차 구현 범위에서 제외한다. 재질과 조명만으로 품질을 먼저 판정한다.

내부 재질·조명 기준은 `physical-3d-renders_260702`의 패키지 렌더 이미지처럼 어두운 스튜디오, 절제된 금속 반사, 선명한 베벨, 통제된 투명감을 목표로 한다. 제품의 레이아웃이나 사물 형태는 복제하지 않는다.

## 7. 30개 오브제 카탈로그

각 행의 오브제는 독립된 모델이다. 괄호 안 영문은 canonical ID다.

| 변수 | 고유 3D 오브제 | 실루엣과 의미 | 짧은 서명 동작 |
|---|---|---|---|
| 기준금리 (`policy_rate`) | 중앙은행 제어 크라운 | 널링 다이얼, 기준 바늘, 낮은 원형 베이스 | 악센트 바늘이 한 눈금 이동 |
| 시장금리 (`market_rate`) | 만기곡선 레일 | 경사진 이중 레일과 이동 캐리지 | 캐리지가 곡선을 따라 짧게 이동 |
| 유동성 (`liquidity`) | 삼중 순환 저장조 | 동심 토러스와 내부 순환 로터 | 내부 로터가 20도 회전 |
| 신용스프레드 (`credit_spread`) | 보정 간극 조 | 마주 보는 두 금속 조와 측정 틈 | 두 조가 미세하게 벌어졌다 복귀 |
| 은행 대출 (`bank_lending`) | 이중 피스톤 신용 펌프 | 평행 실린더와 단방향 게이트 | 악센트 피스톤이 한 번 전진 |
| 소비자물가 (`cpi`) | 가중 가격 인덱스 드럼 | 두께가 다른 분절 링과 기준 눈금 | 드럼이 한 칸 래칫 회전 |
| 기대인플레이션 (`inflation_exp`) | 선행 기대 렌즈 | 열 코어 위에 떠 있는 조준 렌즈 | 렌즈 초점이 앞쪽으로 이동 |
| 임금 (`wages`) | 보상 래칫 실린더 | 계단형 톱니와 누적 드럼 | 톱니가 한 단계 상승 |
| 원·달러 환율 (`fx`) | 쌍축 환율 짐벌 | 서로 반대 방향의 두 비대칭 링 | 두 링이 반대 방향으로 회전 |
| 수출 (`exports`) | 외향 화물 베인 | 중심에서 바깥으로 뻗은 방향성 날개 | 베인이 바깥쪽으로 짧게 전개 |
| 경상수지 (`current_account`) | 유출입 원장 휠 | 양쪽 계수판과 균형축을 가진 휠 | 균형축이 한 번 수평을 찾음 |
| 외국인 자본 유입 (`capital_flows`) | 유입 게이트 매니폴드 | 여러 입구가 중앙 챔버로 모이는 구조 | 입구 게이트가 순서대로 열림 |
| 미국 기준금리 (`fed_rate`) | 외곽 궤도 거버너 | 12분할 대형 다이얼과 궤도 추 | 궤도 추가 반 바퀴 감속 이동 |
| 글로벌 경기 (`global_growth`) | 팽창형 지구 케이지 | 세 직교 밴드와 방사 스트럿 | 밴드가 바깥으로 미세 팽창 |
| 가계 소비 (`consumption`) | 수요 플라이휠 | 꽃잎형 분절 베인이 달린 낮은 회전체 | 베인이 한 차례 수축·복원 |
| 기업 투자 (`investment`) | 전개형 자본 트러스 | 수직 마스트와 접힌 삼각 트러스 | 트러스가 한 단계 펼쳐짐 |
| 고용 (`employment`) | 인력 연동 링크 | 세 수직 기둥을 하나의 링이 연결 | 연결 링이 기둥을 잠금 |
| 기업실적 (`earnings`) | 이익 프리즘 | 베벨 프리즘 안의 상승 계단 구조 | 내부 계단이 한 칸 상승 |
| 부도·연체 (`defaults`) | 파열선 압력용기 | 통제된 균열선과 안전 래치 | 래치가 풀렸다 즉시 복귀 |
| 경제성장 GDP (`gdp`) | 육엽 거시 플라이휠 | 중앙 허브와 여섯 방사 베인 | 여섯 베인이 동시에 확장 |
| 주가 (`stocks`) | 호가창 핀 어레이 | 높이가 다른 수직 핀과 가격 스핀들 | 스핀들이 핀 사이를 상승 |
| 부동산 가격 (`housing`) | 구조 프레임 매스 | 포털 프레임과 층판을 결합한 건축 구조 | 상부 층판이 짧게 상승 |
| 가계부채 (`household_debt`) | 압축 코일 저울 | 무게추가 걸린 균형빔과 조임 코일 | 코일이 반 바퀴 조여짐 |
| 유가 (`oil`) | 산업 압력 캡슐 | 축 배관과 보정 밸브가 붙은 캡슐 | 밸브가 30도 회전 |
| 원자재 (`commodity`) | 원광 코어 도가니 | 가공 프레임 안의 다면체 원광 코어 | 코어가 낮게 한 번 들림 |
| 재정지출 (`fiscal`) | 예산 배분 매니폴드 | 한 입구가 세 보정 출구로 갈라지는 구조 | 세 출구 밸브가 차례로 열림 |
| 지정학 리스크 (`geopolitics`) | 분할 방패 자이로 | 어긋난 두 반구 셸과 긴장축 | 두 셸이 반대 방향으로 비틀림 |
| 기술·생산성 (`tech`) | 칩 격자 마이크로터빈 | 회로 격자와 고속 소형 로터의 결합 | 로터가 짧게 가속 |
| 투자심리 (`risk_sentiment`) | 양극 변동성 진자 | 두 극점 사이 진자와 비틀린 짐벌 | 진자가 한쪽으로 갔다 중심 복귀 |
| 소비자심리 (`consumer_conf`) | 안정 지평계 | 수평 프레임과 중앙 신뢰 베인 | 베인이 상승하고 수평 고정 |

### 중복 방지 기준

- 다이얼 계열인 `policy_rate`, `fed_rate`는 크기만 다르게 만들지 않는다. 전자는 중앙 제어 크라운, 후자는 외곽 궤도 거버너로 구조부터 구분한다.
- 심리 계열인 `risk_sentiment`, `consumer_conf`는 모두 분홍색이지만 전자는 양극 진자, 후자는 안정 지평계로 읽힌다.
- 흐름 계열인 `exports`, `capital_flows`, `current_account`는 각각 외향, 유입, 균형이라는 다른 방향 문법을 가진다.
- 자산 계열인 `stocks`, `housing`, `household_debt`는 핀 배열, 구조 프레임, 코일 저울로 분리한다.
- 원자재 계열인 `oil`, `commodity`는 압력·배관과 원광·도가니로 분리한다.

## 8. 허브 크기 산식

### 8.1 의미

크기는 단순 직접 연결 수가 아니라 **이 변수가 강하고 신뢰도 높은 인과 경로의 중심에 얼마나 놓여 있는가**를 뜻한다. 방향 그래프의 발신 영향력과 수신 의존도를 함께 반영하되, 발신 영향력에 조금 더 무게를 둔다.

### 8.2 계산

각 엣지 `e`의 품질을 다음처럼 0~1 범위로 둔다.

```text
edgeQuality(e) = (strength / 3) × (confidence / 3)
```

노드 `i`에서 `j`로 가는 1~3홉 단순 경로 `p`의 점수는 다음과 같다.

```text
pathScore(p) = product(edgeQuality(e)) × 0.68^(hopCount - 1)
```

- 같은 도착점으로 가는 경로가 여러 개면 가장 강한 경로 하나만 쓴다.
- 경로 안에서 같은 노드를 다시 방문하지 않는다.
- 부호는 허브성 크기에 쓰지 않고 절댓값 연결 품질만 쓴다.
- `outInfluence(i)`는 `i`에서 다른 노드로 가는 최강 경로 점수의 합이다.
- `inExposure(i)`는 다른 노드에서 `i`로 오는 최강 경로 점수의 합이다.
- 두 분포를 각각 최솟값·최댓값 기준으로 0~1 정규화한 뒤 다음처럼 결합한다.

```text
hubRaw(i) = 0.60 × normalize(outInfluence(i))
          + 0.40 × normalize(inExposure(i))
```

전체 노드의 `hubRaw`를 10번째~90번째 백분위에서 절삭하고 다시 0~1로 정규화해 `hubScore`를 만든다. 극단값 하나가 나머지 29개를 지나치게 작게 만들지 않게 하기 위함이다.

### 8.3 화면 크기 매핑

모든 모델의 경계구 반지름을 먼저 1로 정규화한다. 화면에 보이는 면적이 점수에 선형으로 대응하도록 반지름에는 제곱근을 적용한다.

```text
radiusScale = sqrt(0.82^2 + (1.28^2 - 0.82^2) × hubScore)
```

따라서 최소·최대 반지름 비는 약 1.56배이고, 투영 면적 차이는 약 2.44배다. 작은 모델도 클릭 가능 영역은 최소 44 CSS 픽셀 상당의 별도 레이캐스트 프록시로 보장한다.

### 8.4 설명과 범례

- 범례에는 **낮음·중간·높음** 3개 크기 표본을 둔다.
- 변수 상세에는 `인과 허브 78/100`처럼 숫자를 노출한다.
- 도움말에는 직접 연결 수가 아니라 최대 3단계의 강한 경로를 반영한다고 설명한다.
- 동일 점수는 동일 경계구 크기를 가져야 하며, 원본 모델 크기 차이가 결과를 바꾸면 안 된다.

## 9. Blender 저작 규격

### 9.1 도구와 설치

- 서면 스펙 승인 후 설치 시점의 공식 Blender 장기지원(LTS)판을 확인해 설치한다. 목표 계열은 공식 활성 배포판인 [Blender 4.5 LTS](https://www.blender.org/get-involved/Dashboard/)다.
- 대형 실행 파일은 저장소에 넣지 않는다.
- `.blend` 파일을 조형의 정본으로 삼고, 30개 모두 Blender 안에서 개별 하드서피스 모델링과 시각 보정을 거친다.
- Blender 파이썬은 이름·재질·모디파이어 골격 생성과 검증·내보내기를 자동화한다. 최종 형상을 Three.js 프리미티브 조립이나 코드 전용 생성물로 대체하지 않는다.
- 수작업 보정 뒤에도 검증·내보내기 스크립트 하나로 동일한 GLB를 다시 만들 수 있게 한다.

### 9.2 장면 계약

- 각 오브제의 최상위 객체 이름은 canonical ID와 정확히 같아야 한다.
- 원점은 시각적 질량 중심에 둔다.
- 기본 정면은 카메라를 향하는 `+Z`, 위쪽은 `+Y`로 통일한다.
- 로컬 스케일과 회전을 적용한 상태로 내보낸다.
- 최상위 객체 아래 렌더 메시 이름은 기본적으로 `body`, `accent` 두 개다.
- `accent`의 원점은 짧은 서명 동작의 회전·이동 피벗으로 쓴다.
- 보이는 모델과 별개인 충돌 메시, 카메라, 조명은 GLB에 포함하지 않는다.
- 삼각형, 노멀, 접선 정보를 검증하고 겹친 면과 비다양체 형상을 제거한다.

### 9.3 파일 구조

```text
scripts/blender/econ-node-library.blend
scripts/blender/scaffold-econ-node-library.py
scripts/blender/export-econ-node-library.py
scripts/blender/validate-econ-node-library.py
data/models/econ-node-library.glb
vendor/addons/loaders/GLTFLoader.js
vendor/addons/environments/RoomEnvironment.js
vendor/addons/utils/BufferGeometryUtils.js
```

Three.js 코어와 부가 모듈은 모두 현재와 같은 r170으로 맞추고, 부가 모듈의 import 경로는 로컬 `vendor/three.module.js`를 사용하게 고정한다. Three.js 전체 버전 업그레이드는 이번 범위에 포함하지 않는다.

기존 최상위 폴더를 유지하기 위해 Blender 원본·빌드 도구는 `scripts/blender/`, 런타임 모델은 `data/models/`에 둔다.

### 9.4 모델 예산

- 모델당 1,500~3,000 삼각형을 목표로 한다.
- 전체 45,000~90,000 삼각형 범위로 제한한다.
- 모델당 렌더 프리미티브는 원칙적으로 2개, 예외적으로 3개까지 허용한다.
- 단일 GLB는 2MB 이하를 목표로 하고 3MB를 하드캡으로 둔다.
- 텍스처 없이 재질과 기하로 해결하는 것을 우선한다.
- 텍스처가 반드시 필요하면 공용 1,024픽셀 패킹 텍스처 1장까지만 쓴다.
- 반복되는 베어링, 링, 축의 세그먼트 수는 화면 크기를 기준으로 줄인다.

## 10. Three.js 런타임 구조

### 10.1 점진적 교체

현재 구체 노드는 즉시 제거하지 않는다.

1. 기존 구체로 장면과 상호작용을 먼저 초기화한다.
2. `GLTFLoader`가 단일 GLB를 비동기로 불러온다.
3. 30개 canonical ID와 GLB 최상위 이름을 검증한다.
4. 유효한 모델만 경계구를 정규화하고 해당 구체와 교체한다.
5. 누락·손상 모델은 기존 구체를 계속 사용한다.

이 구조는 느린 로딩이나 자산 오류가 앱 전체를 막는 일을 방지한다.

### 10.2 객체 계층

```text
nodeRoot                 위치, 허브 크기, 레버 드래그
├─ modelRoot             경계구 정규화, 기본 방향
│  ├─ body               공통 PBR 몸체 재질
│  └─ accent             범주색, 짧은 서명 동작
├─ stateRing             선택·시나리오 방향·크기
├─ hitProxy              보이지 않는 최소 클릭 영역
└─ labelAnchor           기존 2D 라벨 연결점
```

GLB의 재질은 이름과 슬롯만 전달한다. 런타임에서 공용 몸체 재질과 9개 범주 인레이 재질로 교체해 재질 수와 상태 드리프트를 줄인다.

### 10.3 데이터 흐름

```text
NODES + EDGES
  → hubScore 계산
  → spatial position 보존
  → GLB canonical ID 매칭
  → bounding sphere 정규화
  → hub radiusScale 적용
  → 선택·시나리오·전파 상태 오버레이
```

허브 점수 계산은 화면 코드와 분리된 순수 함수로 구현하고, 30개 결과를 테스트에서 스냅샷으로 고정한다.

### 10.4 엣지 렌더 예산 회수

현재 107개 엣지가 개별 객체로 렌더링된다. 3D 모델을 추가하기 전에 비활성 엣지를 부호와 확실성 기준의 소수 배치로 합치는 방식을 검토한다. 선택·전파 중인 엣지는 기존 개별 애니메이션 경로를 유지할 수 있다. 이 최적화는 모델 품질을 낮추지 않고 드로콜 예산을 확보하기 위한 것이다.

## 12. 라벨과 접근성

- 변수명은 기존 2D 라벨 체계를 유지한다.
- 모델 내부에 정보 전달용 텍스트를 넣지 않는다.
- 색과 형태를 보지 못해도 변수명, 범주, 허브 점수, 시나리오 방향을 상세 패널에서 읽을 수 있어야 한다.
- 작은 모델의 시각 크기와 클릭 크기를 분리한다.
- 키보드 포커스와 선택 상태는 3D 움직임 없이도 명확해야 한다.
- `prefers-reduced-motion`과 앱의 모션 토글을 모두 존중한다.
- 모델 로딩 실패는 스크린리더 흐름이나 키보드 탐색을 바꾸지 않는다.

## 14. 대표 6개 검증 게이트

먼저 다음 6개를 만든다.

1. `policy_rate`, 작은 중앙 다이얼 계열
2. `fx`, 상반 회전 짐벌 계열
3. `oil`, 산업 압력 캡슐 계열
4. `housing`, 건축 프레임 계열
5. `gdp`, 중심 플라이휠 계열
6. `risk_sentiment`, 심리 진자 계열

이 조합은 원형, 프레임, 캡슐, 짐벌, 방사형, 진자형 실루엣과 6개 역할 범주를 동시에 검증한다.

### 통과 조건

- 6개를 회색 단색으로 봐도 서로 구분된다.
- 일반 카메라 거리에서 주 실루엣과 악센트가 읽힌다.
- 범주색 면적이 물체 전체의 약 10~20%를 넘지 않는다.
- 베벨과 곡률이 환경광에 반응하고, 와이어프레임 없이도 입체가 읽힌다.
- 어느 모델도 클립아트식 집, 물방울, 동전, 사람 모형처럼 보이지 않는다.
- 모델당 삼각형과 프리미티브 수가 예산 안에 든다.
- 실제 지도에서 라벨과 엣지를 가리지 않는다.
- 선택, 전파, 모션 감소 상태가 모두 작동한다.
- 데스크톱과 390px 모바일 검증에서 콘솔 오류가 없다.

대표 6개가 통과하기 전에는 나머지 24개를 대량 제작하지 않는다. 이 게이트가 전체 30개의 조형 품질과 성능을 결정하는 시작 산출물이다.

## 15. 오류 처리와 되돌림

- GLB 요청 실패: 기존 구체 30개를 유지하고 앱을 계속 실행한다.
- 일부 canonical ID 누락: 누락 모델만 구체를 유지하고 하나의 집계 경고를 기록한다.
- 중복 ID: 중복 모델을 사용하지 않고 첫 유효 모델만 채택한다.
- 경계구가 0이거나 유한하지 않음: 해당 모델을 거부하고 구체로 대체한다.
- 재질 컴파일 실패: 해당 메시를 공용 티타늄 재질로 대체한다.
- 성능 예산 초과: 텍스처 품질보다 세그먼트·프리미티브·비활성 엣지 드로콜을 먼저 줄인다.
- 모바일 성능 저하: 픽셀 비율 1.5 제한, 렌즈 재질 단순화, 서명 동작 축소 순으로 완화한다.
- 배포 회귀: 구체 폴백 플래그로 GLB 교체만 끌 수 있게 한다.

## 16. 검증 계획

- 허브 점수 30개가 0~1 범위이고 동일 입력에서 결정적으로 재생성되는지 검사한다.
- `git diff --check`와 기존 정적 데이터 검증을 통과시킨다.

### 실브라우저 검증

- 홈, 변수 선택, 원인 보기, 결과 보기, 시뮬레이터, 사례, 루프를 순회한다.

- 데스크톱과 390px 모바일에서 스크린샷을 비교한다.
- `renderer.info.render.calls`, 삼각형 수, 프레임 시간을 기록한다.
- 느린 네트워크와 GLB 404를 강제로 만들어 폴백을 확인한다.
